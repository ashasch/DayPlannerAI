# AI Day Planner — Етап 1

Застосунок, у який користувач «вивантажує» все, що в голові, текстом або голосом,
а AI структурує це в задачі та план дня.

Етап 1 закладає фундамент: структуру проєкту, авторизацію, локалізацію, екран
Capture і робочий AI-розбір тексту в структуровані задачі (через OpenRouter).

## Швидкий старт

```bash
npm install

# Підтягнути змінні з Vercel (включно з POSTGRES_URL):
npx vercel link
npx vercel env pull .env.local

npm run db:migrate   # створити таблиці
npm run dev
```

Відкрий http://localhost:3000.

### Локальна база

Найпростіший варіант — **та сама база Vercel Postgres**: нічого не треба
встановлювати, а поведінка локально й у продакшні збігається один в один.
`vercel env pull` кладе `POSTGRES_URL` у `.env.local`, і все працює.

Якщо потрібна ізольована локальна база (щоб не чіпати продакшн-дані), підніми
Postgres у Docker і вкажи його URL у `DATABASE_URL`:

```bash
docker run --name dayplanner-pg -e POSTGRES_PASSWORD=postgres -p 5432:5432 -d postgres:17
# DATABASE_URL=postgres://postgres:postgres@localhost:5432/postgres
npm run db:migrate
```

SQLite не підійде: продакшн на Postgres, а розбіжність діалектів (типи `uuid`,
`timestamptz`, поведінка унікальних індексів) ловиться тільки на проді.

### Змінні середовища

| Змінна                          | Обов’язкова | Призначення                                                                    |
| ------------------------------- | ----------- | ------------------------------------------------------------------------------ |
| `DATABASE_URL` / `POSTGRES_URL` | так         | Рядок підключення до Postgres. На Vercel додається автоматично                 |
| `AUTH_SECRET`                   | так         | Ключ підпису сесій Auth.js. Згенерувати: `npx auth secret`                     |
| `OPENROUTER_API_KEY`            | ні\*        | Ключ OpenRouter (`sk-or-…`). Без нього застосунок працює, AI повідомляє про це |
| `OPENROUTER_MODEL`              | ні          | Модель (за замовчуванням `anthropic/claude-sonnet-4.5`)                        |
| `GITHUB_CLIENT_ID` / `_SECRET`  | ні          | Вхід через GitHub. Потрібні **обидві**, інакше провайдер не реєструється       |
| `GOOGLE_CLIENT_ID` / `_SECRET`  | ні          | Вхід через Google. Потрібні **обидві**                                         |

Callback URL, які треба прописати у провайдерів:
`{origin}/api/auth/callback/github` і `{origin}/api/auth/callback/google`
(локально `origin` = `http://localhost:3000`).

`DATABASE_URL` має пріоритет над `POSTGRES_URL` — так локальне перевизначення
не конфліктує зі змінною, яку інжектить Vercel.

\* Ключ читається **виключно** з `OPENROUTER_API_KEY` і ніде не хардкодиться.
Модулі, що його використовують, імпортують `server-only`, тож випадковий імпорт
з клієнтського компонента впаде на етапі збірки, а не витече в браузер.

**Провайдер — OpenRouter, не Anthropic напряму.** Транспортом лишається
`@anthropic-ai/sdk`, бо OpenRouter віддає Anthropic-сумісний `/v1/messages` з
рідною формою відповіді — тож типізований SDK працює без шару трансляції.
Два наслідки, на які легко натрапити:

- `baseURL` — це `https://openrouter.ai/api` **без** `/v1`: SDK сам дописує
  `/v1/messages`, інакше вийде `/api/v1/v1/messages` і 404.
- ID моделі має бути з неймспейсом OpenRouter (`anthropic/claude-sonnet-4.5`);
  голий `claude-…` відхиляється.

### Архітектурні рішення

**Локалізація без перезавантаження.** Мова зберігається в cookie
(`dayplanner.locale`, 1 рік), а не в URL. Перемикач викликає server action і
`router.refresh()` — серверне дерево перемальовується з новими повідомленнями,
клієнтський стан (зокрема текст у Capture) зберігається. Українська — мова за
замовчуванням; за відсутності cookie працює узгодження за `Accept-Language`.

**Авторизація.** Auth.js v5, JWT-сесії. Конфіг розділено на `auth.config.ts`
(Edge-safe, для middleware) і `auth.ts` (Node runtime, з провайдерами).
Провайдери: Email/пароль (Credentials), GitHub, Google.

**OAuth реєструється умовно.** Провайдер додається в масив тільки якщо є
**обидві** половини пари (`*_CLIENT_ID` + `*_CLIENT_SECRET`). Auth.js падає при
створенні напівналаштованого провайдера, а це поклало б увесь роут `/api/auth`
— разом зі входом по паролю — на будь-якому середовищі без OAuth-ключів.
Сторінки входу/реєстрації рендеряться на сервері й передають у кнопки реальний
стан, тож кнопка не почне редірект, який гарантовано впаде на колбеку.

**Прив'язка OAuth-акаунта до нашої таблиці.** Для OAuth `user.id` — це
ідентифікатор GitHub/Google, а не наш uuid. Якби він потрапив у токен, перевірка
в `app/(app)/layout.tsx` не знайшла б користувача і викинула б щойно залогіненого
назад через `stale-session`. Тому `jwt`-колбек перевизначено в `auth.ts` (він
читає БД, чого Edge-middleware не може) — він знаходить або створює локальний
акаунт за email і кладе в токен **наш** uuid.

Акаунти зіставляються за email, тому вхід через GitHub і потім через Google веде
до тих самих задач. Зіставлення виконується **лише для підтверджених адрес**
(`email_verified` у Google; GitHub віддає верифіковану primary-адресу через
скоуп `user:email`). Інакше будь-хто міг би створити OAuth-акаунт з чужим email
і перехопити акаунт із паролем.

OAuth-акаунт створюється з `passwordHash = null` — пароль можна встановити
пізніше через флоу відновлення.

**Сховище.** Postgres через Drizzle ORM. `UserRepository` /
`PasswordResetTokenRepository` — інтерфейси в `lib/db/types.ts`, реалізація в
`lib/db/postgres-repository.ts`, підключення в `lib/db/index.ts`. Заміна рушія =
одна нова реалізація інтерфейсу; жоден виклик не змінюється.

З’єднання створюється **ліниво**, при першому запиті, а не при імпорті модуля.
`next build` імпортує всі роут-модулі, щоб зібрати page data — при жадібному
підключенні збірка сама б вимагала доступної бази, і відсутній `DATABASE_URL`
ламав би build замість того, щоб дати зрозумілу помилку на тому одному запиті,
якому база справді потрібна.

Пул налаштовано під serverless: `max: 1` (кожен інстанс Vercel — окремий процес,
інакше ліміт з’єднань вичерпується зі зростанням конкурентності) і
`prepare: false` (Vercel Postgres і Supabase ставлять перед базою PgBouncer у
transaction mode, який не вміє переносити server-side prepared statements між
пулованими з’єднаннями).

**Гонка при реєстрації.** Перевірка «чи є такий email» перед вставкою — це
read-then-write, і дві одночасні реєстрації на одну адресу її проходять обидві.
Вирішує унікальний індекс на `users.email`: репозиторій ловить `23505` і кидає
`EmailAlreadyExistsError`, а сервісний шар перекладає її в `EMAIL_TAKEN`.
Попередня перевірка лишилась як швидкий шлях — щоб не витрачати ~300 мс на bcrypt
для наперед приреченої вставки.

### Міграції

```bash
npm run db:generate   # згенерувати SQL зі змін у lib/db/schema.ts
npm run db:migrate    # застосувати міграції
npm run db:studio     # переглянути дані
```

Міграції лежать у `drizzle/` і комітяться в репозиторій.

**Пошта.** `Mailer` з консольним транспортом: посилання для відновлення пароля
друкується в консоль сервера. Для продакшну — нова реалізація інтерфейсу.

**Безпека паролів.** bcrypt (cost 12). Токени відновлення зберігаються тільки як
SHA-256 хеші, одноразові, живуть годину. `/forgot-password` відповідає однаково
для наявних і відсутніх адрес, щоб не давати перелічувати акаунти.

## Що перевірено

Наскрізно в браузері: реєстрація → автовхід → Capture; вхід (правильний і
хибний пароль); повний цикл відновлення пароля з перевіркою одноразовості
токена; захист маршрутів через middleware з `callbackUrl`; перемикання UA/EN без
перезавантаження зі збереженням набраного тексту; автовисота textarea; кнопка
очищення; мобільний вьюпорт 375px без горизонтального скролу; стан-машина
мікрофона (idle → recording → помилка → idle) з локалізованим toast; протухла
сесія (JWT є, користувача вже немає) коректно розлогінюється без redirect-циклу.

**AI-аналіз перевірено на реальному виклику OpenRouter:** брудний список
українською розібрано на задачі з коректними пріоритетами (звіт до п'ятниці →
high, «колись розібрати шафу» → low) і коректними датами, Zod-валідація пройшла,
заголовки повернулись українською.

**Календар і Вхідні перевірено наскрізь на реальній Neon Postgres:** збереження
чернеток → бейджі дат у «Вхідних» → ручна зміна дати → тижнева й місячна сітки →
навігація по періодах → drag & drop між днями (з перевіркою, що дата справді
записалась у БД) → діалог деталей. Плюс негативні шляхи API: чужий/неіснуючий
`id` → 404, неможлива дата `2026-02-31` → 422, порожній patch → 422.

**Не перевірено наживо:** реальне розпізнавання мовлення (у тестовому браузері
мікрофон заблоковано — перевірено лише коректну обробку відмови); **операції з
Postgres** (у середовищі розробки не вдалося підняти локальну БД); **рендер
списку задач у браузері** — сторінка Capture за авторизацією, а вона потребує
БД, тож `TaskList` перевірено лише типами й збіркою, не візуально.

Після переходу на Postgres перевірено: типізація, `npm run build` **без**
`DATABASE_URL` (лінива ініціалізація працює), рендер сторінок у `npm run dev`,
і що відсутній `DATABASE_URL` дає зрозумілу помилку в логах сервера та
загальний `unknown` клієнту, не роняючи процес.

## Екрани

**Захоплення** — вивантаження думок текстом/голосом. AI розбирає текст на
задачі; результат показується як **чернетка**: можна прибрати зайве й змінити
дату, і лише тоді «Зберегти у Вхідні». Нічого не потрапляє в базу до підтвердження.

**Вхідні** — усі збережені задачі, згруповані на «Заплановані» (за датою) і «Без
дати». Дата міняється прямо з картки.

**Календар** — тижнева сітка (7 колонок) і місячна (6×7). Навігація стрілками,
кнопка «Сьогодні», перемикач Тиждень/Місяць. Задачу можна перетягнути на інший
день або клікнути, щоб відкрити деталі. Задачі **без** `plannedDate` у календар
не потрапляють — вони лишаються тільки у «Вхідних».

### Дати

`planned_date` — колонка `date`, не `timestamp`. Це календарний день: «зробити
20-го» означає 20-те там, де перебуває користувач, а збереження моменту часу
зсувало б задачу на сусідній день між таймзонами й на переходах DST. Уся
арифметика дат — у `lib/date.ts` і працює в локальній зоні; `new Date('2026-07-25')`
навмисно ніде не використовується, бо парситься як UTC-північ і в половині світу
показує 24-те.

Клієнт передає своє «сьогодні» в `/api/ai/capture`: на Vercel сервер живе в UTC,
тож інакше «завтра» пізно ввечері резолвилося б у неправильний день.

## AI-аналіз

`lib/ai/capture.ts` розбирає текст на структуровані задачі: `title`,
`priority` (high/medium/low), `category`, `estimatedMinutes`. Відповідь моделі
проганяється через Zod-схему (`lib/ai/schemas.ts`), тож зіпсований або
згаллюцинований JSON стає спійманою помилкою, а не битими даними в UI.

Модель просять віддати голий JSON, а відповідь префіксують `{` — це найдешевший
спосіб не отримати markdown-огорожу чи пояснення навколо JSON. Парсер додатково
має фолбек на пошук зовнішніх дужок.

**Розпізнавання дат.** «завтра», «в п'ятницю», «20 числа», «5 серпня» →
`plannedDate`. Нерозпізнане («колись», «цього тижня») лишається `null` — модель
явно інструктована не вгадувати.

Дві речі, без яких це не працює надійно, обидві знайдені на реальних прогонах:

- **Таблиця дат у промпті.** Самої лише «сьогодні N» замало: модель правильно
  називає день тижня, але помиляється на 1 у даті, бо це багатокрокова
  арифметика. У промпт підставляється готовий список на 21 день
  (`2026-07-24 = Friday`), і задача зводиться до пошуку в таблиці.
- **`temperature: 0`.** За замовчуванням API дає 1.0, і та сама «до п'ятниці»
  один раз резолвилась у п'ятницю, а наступного разу в суботу.

Після обох: 3 прогони поспіль дали ідентичний і повністю правильний результат
на 8 різних формулюваннях дат.

## Далі

Задачі зберігаються, плануються по днях і показуються в календарі. Не зроблено:
редагування назви/пріоритету (зараз редагується тільки дата), позначка
«виконано», повторювані задачі, drag & drop з «Вхідних» у календар.
